# Secure E2EE Chat Application

A full-stack real-time secure chat application implementing End-to-End Encryption (E2EE) using hybrid cryptography (AES-GCM for messages and RSA-OAEP for key exchange).

## Features
- **End-to-End Encryption**: True E2EE, server never sees the unencrypted message.
- **Hybrid Cryptography**: AES-256 for symmetric message encryption layer and RSA-2048 for asymmetric key exchange.
- **Real-Time Messaging**: Built on WebSockets via `socket.io`.
- **JWT Authentication**: User authentication and session management using tokens.
- **Modern UI**: Clean and minimal interface resembling WhatsApp, built with React and Tailwind CSS.
- **No Plaint-text Storage**: Only bcrypt hashed passwords and encrypted payloads are stored/relayed.

## Prerequisites
- Node.js (v18+)
- MongoDB running locally or on MongoDB Atlas cluster.

## Installation & Setup

### 1. Backend Setup

Open a terminal and navigate to the `server` directory:

```bash
cd server
npm install
```

Configure Environment variables by making sure `.env` inside `server` looks like this:

```
PORT=5000
MONGO_URI=mongodb://127.0.0.1:27017/e2ee-chat
JWT_SECRET=super_secret_jwt_key_change_in_production
```

Run the backend server:

```bash
npm run dev
# Server should now run on http://localhost:5000
```


### 2. Frontend Setup

Open another terminal and navigate to the `client` directory:

```bash
cd client
npm install
```

Make sure the `.env` inside `client` specifies the correct backend URL:

```
VITE_API_URL=http://localhost:5000
```

Start the Vite dev server:

```bash
npm run dev
# Application will run on http://localhost:5173
```


## How the Encryption Works
The encryption fully happens within the browser's `Web Crypto API`, adhering to an E2EE specification:
1. **Key Generation**: When a user logs in, a new `RSA-OAEP` public/private key pair is generated by their browser.
2. **Key Sharing**: The user's public key is sent to the server to be shared with other clients; the private key never leaves the device.
3. **Drafting a Message**: A user writes a message to an online contact.
4. **Symmetric Encryption**: The browser generates a random one-time `AES-GCM` key and an Initialisation Vector (IV). It encrypts the message with this AES key.
5. **Asymmetric Key Exchange**: The browser fetches the recipient's `RSA` Public Key from the directory and uses it to encrypt the newly forged `AES-GCM` key.
6. **Relay Payload**: The cipher payload (Encrypted Message, Encrypted AES Key, and IV) is sent over websockets to the recipient.
7. **Local Decryption**: The recipient receives it. They decrypt the AES key using their `RSA Private Key`. Finally, they use the decrypted AES key and IV to decrypt the cipher message text.
